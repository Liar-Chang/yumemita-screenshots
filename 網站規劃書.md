## 2. 參考網站逆向分析（2026-07-14 實測）

我實際操作並拆解了 ave-mujica-images.pages.dev 的 JS bundle，架構如下：

| 面向 | 參考站做法 |
|---|---|
| 前端 | React + Material UI 的純靜態 SPA（Vite 打包），結果區用虛擬化清單 |
| 資料模型 | 每張圖一筆 `{name: "台詞文字", episode: 集數}`，**圖片檔名就是台詞**（如 `CRYCHIC已經不在了-xxx.webp`） |
| 資料存放 | **沒有後端、沒有資料庫、沒有 API**——約 4,785 張 WebP 連同索引全部打包成靜態檔 |
| 搜尋 | 純前端子字串比對（輸入「為什麼」→ Mujica 22 筆 / MyGO 19 筆，零網路請求） |
| 圖片密度 | 兩部各 13 集共 ~4,785 張 ≈ **每集約 184 張**，約等於「每句台詞一張」 |
| 託管 | Cloudflare Pages（免費） |

**結論**：這個架構完全可以複製，而且成本為零。但有兩點我們要做得更好：

1. **不要用「檔名＝台詞」**：台詞含 `?" <>|` 等字元在 Windows 會炸檔名、同台詞會撞名、長台詞會超過路徑長度限制。改用 `e01/0042.webp` + JSON 索引，還能額外存時間戳、角色標籤。
2. **索引與圖片分離**：圖片放 `public/`，索引是建置時產生的 JSON（約 5,000 筆 ≈ gzip 後 100 多 KB，前端一次載入毫無壓力）。

## 3. 關鍵決策：截圖密度（★ 回來後請選一個）

「逐幀」字面上是每一幀都截，先看真實數字（單集正片約 24 分鐘）：

| 方案 | 做法 | 13 集總量 | 容量估算 | 託管 |
|---|---|---|---|---|
| **A. 每句台詞 1 張**（參考站等級） | 取每句字幕時間中點抽 1 幀 | ~4,500 張 | ~270 MB | Pages 免費 ✅ |
| **B. 每句 1 張＋場景補圖**（建議 ★） | A ＋ 對無台詞片段做場景變化偵測補圖、長句抽 2~3 張 | ~9,000 張 | ~550 MB | Pages 免費 ✅ |
| C. 固定每 0.5 秒 1 張 | 不依賴字幕的暴力採樣 | ~37,000 張 | ~2.2 GB | 超過 Pages 兩萬檔上限 ❌ → 需 R2 |
| D. 真・逐幀（23.976fps） | 每幀都存（可先用 pHash 去除重複幀，動畫大量停格可砍 5~7 成） | 去重後仍 ~15 萬張 | 5~9 GB | 需 R2（免費額度 10GB 內，勉強） |

**我的建議：方案 B。** 理由：搜尋的最小單位本來就是「一句台詞」，方案 C/D 會讓同一句話出現幾十張近乎相同的圖，搜尋體驗反而變差、成本暴增。若之後真想要逐幀，架構上留了升級路（見 §5 的「同句多幀」設計），可以只對名場面加密度。

## 4. 素材需求（★ 你需要準備的東西）

網站本身不碰影片，只吃「處理管線」產出的圖＋文字。管線需要每集兩樣東西：

1. **影片檔**（mp4/mkv，1080p 佳）——請自行以合法方式取得（自錄、BD 等）。動畫瘋直接下載違反其服務條款，這部分我不會代勞，規劃以「你已備妥檔案」為前提。
2. **台詞文字＋時間軸**，兩條路：
   - **路線甲：外掛字幕檔（.ass/.srt）**——若手上的影片來源附獨立字幕軌，直接解析，全自動，零人工。
   - **路線乙：內嵌字幕 OCR**——動畫瘋的繁中字幕是壓在畫面上的。用 VideoSubFinder 抓字幕出現時間軸＋裁切區域，再用 PaddleOCR（繁中模型）辨識文字，最後人工校對錯字（實測這類流程每集約需 15~30 分鐘人工）。台詞會與官方翻譯一字不差，做迷因梗圖搜尋反而是最理想的來源。

另外需要（都免費）：**GitHub 帳號**（版控＋自動部署）、**Cloudflare 帳號**（Pages 託管）。

## 5. 系統架構

```
┌─ 素材/（不進版控）
│    ep01.mkv, ep01.ass ...
│
├─ pipeline/（Python 處理管線）
│    extract.py      解析字幕 → ffmpeg 抽幀 → 壓 WebP → 產生/合併 JSON 索引
│    ocr/            (路線乙才需要) VideoSubFinder + PaddleOCR 流程
│
└─ site/（Vite + React + TypeScript + Tailwind 前端）
     public/img/e01/0042.webp        圖片（1280×720 WebP q80，約 60KB/張）
     public/index/yumemita.json      搜尋索引
     src/ ...                        UI
```

**索引格式**：

```json
{
  "updated": "2026-07-14",
  "items": [
    { "id": "e01-0042", "ep": 1, "t": 312.4,
      "text": "台詞內容", "img": "img/e01/0042.webp", "tags": ["阿拉蕾"] }
  ]
}
```

**前端功能規格**

- MVP：關鍵字即時搜尋（子字串比對＋全半形正規化，選配簡→繁轉換讓簡體輸入也搜得到）、集數下拉篩選、時間軸排序、瀑布流縮圖（虛擬化捲動＋lazy load）、點圖開大圖 Lightbox（顯示集數/時間/完整台詞）、**一鍵複製圖片到剪貼簿**、複製分享連結（URL 帶查詢參數可直連）、RWD 手機版。
- 加值（上線後看需求加）：角色標籤篩選、「隨機一張」、前一句/後一句導覽、同句多幀檢視（逐幀升級路）、多幀合成 GIF、深色模式、PWA。
- 頁尾：版權聲明（非營利粉絲網站、著作權歸 ©BanG Dream! Project / bushiroad，接獲權利方要求即配合下架）＋聯絡方式。

**無台詞片段的截圖策略（三層，已實作於 pipeline/extract.py）**

1. 台詞層：每句字幕抽 1 張（超過 6 秒的長句平均抽 2~3 張）。
2. 場景層：無台詞空窗（≥4 秒）用 ffmpeg 場景變化偵測，每個鏡頭切換抓一張（最小間隔 2.5 秒）——OP/ED 與純演出段落主要靠這層。
3. 保底層：連場景變化都沒有的長鏡頭（≥6 秒），每 6 秒固定抽一張。

密度參數都寫在腳本開頭常數區，跑完第 1 集實片後再校準。無台詞圖沒有可搜尋的文字，檢索方式：預設掛「場景」標籤＋集數瀏覽；名場面可事後補描述（參考站的做法是括號命名如「(Mortis踢腿)」），也可以由 Claude 在每週更新時看圖自動寫短描述進索引。

**部署**：GitHub repo 連 Cloudflare Pages，push 即自動建置部署。方案 A/B 的量（<1GB、<2 萬檔）在免費額度內；方案 C/D 才需要把圖搬去 R2。

## 6. 開發階段

| 階段 | 內容 | 預估 |
|---|---|---|
| P0 素材準備 | 你備妥已播出集數的影片（＋字幕檔若有）；開 GitHub / Cloudflare 帳號 | 你的時間 |

| P2 網站 MVP | 搜尋＋集數篩選＋圖牆＋Lightbox＋複製圖片 | 1 天 |

| P4 回填＋週更 | 批次處理已播集數；之後每週四播完跑一次腳本→push（每集機器 10 分鐘＋OCR 校對 15~30 分鐘） | 每週例行 |
| P5 加值功能 | 從加值清單挑著做 | 隨需求 |

素材齊的話，**MVP 上線約 2 個工作天**。

## 7. 風險與注意事項

- **版權**：截圖屬二次利用，Bushiroad 對迷因文化向來寬容（MyGO 梗圖滿天飛），但本質仍是灰色地帶——非營利、標明出處、保留下架管道是底線。影片與字幕取得方式請維持合法。
- **OP/ED**：照常收錄（2026-07-14 已確認）。有歌詞字幕就掛歌詞文字，無字幕的畫面由場景偵測涵蓋。
- **重複台詞**：「嗯」「欸」這類短句會有幾十筆，UI 上照集數/時間排序即可，不需去重（有時就是要找特定那一幕的「嗯」）。
- **Cloudflare Pages 限制**：單次部署 2 萬檔、單檔 25MB——方案 A/B 安全，C/D 必須改 R2。
- **OCR 錯字**：路線乙的辨識準確率約 97~99%，錯字會直接影響搜尋命中，校對步驟不能省。
